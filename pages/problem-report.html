flow ของ addons budget รายงานปัญหา

Summary

Login เข้าแล้ว Budget ทำงานทันที

เราใส่ INSERT INTO "UAT_JBF"."NDBS_LOGIN_TRACE" ไว้ใน NDBS_BUDGET_CONTROL และ NDBS_UpdateAllBudgetAmount เพื่อดูว่า add-on ถูกเรียกเมื่อใด จากนั้นล้าง table แล้ว login เข้า SAP โดยยังไม่เปิดเอกสารหรือเริ่ม transaction ใด ๆ แต่ใน NDBS_LOGIN_TRACE กลับมีข้อมูลถูก insert เข้ามาทันที จึงยืนยันได้ว่า Budget ถูกเรียกตั้งแต่ช่วง login จริง

สิ่งที่พิสูจน์ได้จากการทดสอบนี้

  • แค่ login เข้า SAP ก็มีการเรียก NDBS_BUDGET_CONTROL
  • เมื่อ NDBS_BUDGET_CONTROL ถูกเรียก จะมี NDBS_UpdateAllBudgetAmount ตามมาทันที
  • ทั้งหมดนี้เกิดขึ้นทั้งที่ผู้ใช้ยังไม่ได้เริ่มสร้างเอกสารหรือทำ transaction ใด ๆ
เปิดดูโค้ดและ trace ที่ใช้ยืนยัน

Slow Flow

Flow ที่ทำให้ SAP ช้า

ประเด็นสำคัญมีแค่เส้นทางนี้: Login -> NDBS_BUDGET_CONTROL -> NDBS_UpdateAllBudgetAmount -> for loop ทั้งชุด

1. NDBS_BUDGET_CONTROL ทำอะไร

ดูโค้ด 
if :error = 0 then
  IF ( :object_type in ('30','22','21','20','18','19')) then
    ...
    INSERT INTO "NDBS_BGC_OBDE" ...
    INSERT INTO "NDBS_BGC_OBPE" ...
    Call NDBS_UpdateBudgetAmount(...)
  end if;
end if;

ตัวนี้เป็น budget control หลักของระบบ เมื่อถูกเรียกจะวิ่งแตก logic ตามประเภทเอกสาร แล้วเขียน movement ลงตาราง budget เช่น NDBS_BGC_OBDE และ NDBS_BGC_OBPE พร้อมอัปเดตยอด budget บางส่วนระหว่างทาง

2. เมื่อ NDBS_BUDGET_CONTROL ทำงานก็จะไปเรียกใช้ NDBS_UpdateAllBudgetAmount ต่อ

ดูโค้ด 
DELETE FROM NDBS_BGC_OBPE WHERE "Amount"=0;
DELETE FROM NDBS_BGC_OBDE WHERE "Amount"=0;
call NDBS_UpdateAllBudgetAmount;

แปลว่าเมื่อ budget control ทำงานเสร็จ ระบบจะไม่หยุดแค่นั้น แต่จะไปเรียก NDBS_UpdateAllBudgetAmount ต่อทันที เพื่อคำนวณงบใหม่ทั้งก้อน จึงทำให้ช่วง login กลายเป็นงานหนักทันที

3. NDBS_UpdateAllBudgetAmount เป็นงานหนัก

ดูโค้ด 
Declare cursor loopbudgetdept for Select * From "@NDBS_BGC_BDPL";
Declare cursor loopbudgetproj  for Select * From "@NDBS_BGC_BPJL";

for currloop as loopbudgetdept do
...
end for;

for currloop as loopbudgetproj do
...
end for;

procedure นี้จะวนทีละแถวผ่าน budget department และ budget project ทั้งชุด จากนั้นไป SUM ยอดจาก movement tables แล้ว UPDATE budget master ทีละแถว จึงช้ามากเมื่อข้อมูลเยอะ และยิ่งหนักขึ้นอีกถ้ามีหลาย user login พร้อมกัน

สรุปคือ login ไม่ได้เป็นแค่ขั้นตอนเข้าใช้งานธรรมดา แต่ถูกพาไปทำ budget processing ทันที และตอนจบยังโดนสั่ง full refresh ของงบทั้งระบบผ่าน NDBS_UpdateAllBudgetAmount ที่ใช้ for loop วนข้อมูลทั้งชุด จึงเป็นเหตุผลหลักที่ทำให้ SAP ช้า

Wrong SQL

SQL ที่ยังผิดและทำให้ข้อมูล budget เพี้ยนได้

เหลือแค่จุดที่สำคัญจริงและกระทบข้อมูลโดยตรง

Wrong Query

NDBS_UpdateAllBudgetAmount.sql

WHERE "Project" = currloop."U_Project"
AND "BudgetGroup" = currloop."U_GroupCode"
AND "BudgetYear" = currloop."U_Project"

เอา project code ไปเทียบกับ BudgetYear โดยตรง ทำให้ยอด budget ฝั่ง project เสี่ยงผิดได้ทันที

Wrong Query

NDBS_UpdateBudgetAmount.sql

WHERE "Project" = currloop."U_Project"
AND "BudgetGroup" = currloop."U_GroupCode"
AND "BudgetYear" = currloop."U_Project"

helper ที่ควร update เฉพาะจุดก็ยังใช้ key ผิดแบบเดียวกัน จึงเสี่ยงเพี้ยนซ้ำแม้ไม่วิ่ง full refresh

ข้อสรุปสุดท้ายมีแค่ 2 เรื่อง: Login ทำให้ Budget ทำงานทันทีและลาก full refresh จน SAP ช้า และ query ฝั่ง project ใช้ key ผิด จึงทำให้ข้อมูล budget เพี้ยนได้

ทำไมพอทีม IT แก้ NDBS_UpdateAllBudgetAmount.sql แล้วอาการช้าจึงหายจริง

เพราะจุดที่ถูกแก้คือคอขวดหลักที่ถูกเรียกต่อจาก NDBS_BUDGET_CONTROL ตอน login โดยตรง เดิม procedure นี้ใช้วิธีวน for loop ผ่าน budget ทั้งชุด แล้วค่อย SUM และ UPDATE ทีละแถว แต่ version ที่ทีม IT ปรับใหม่เปลี่ยนเป็นการ UPDATE แบบ set-based ครั้งใหญ่ไม่กี่ครั้งแทน

แบบเดิม ช้าเพราะทำงานแบบทีละแถว

ใช้ cursor สองชุดวิ่งผ่าน budget department และ budget project ทั้งหมด แล้วในแต่ละรอบยังไป SELECT SUM และ UPDATE ซ้ำอีก ทำให้เกิดงานจำนวนมากแบบ row-by-row

แบบใหม่ เร็วขึ้นเพราะรวมงานเป็น bulk update

rewrite ให้เหลือการ UPDATE แบบ set-based สองก้อนใหญ่แทนการวนทีละแถว ทำให้ HANA ประมวลผลเป็นชุดเดียวและลด overhead จาก loop ซ้ำ ๆ ลงอย่างมาก

ดังนั้นเมื่อแก้ NDBS_UpdateAllBudgetAmount.sql แล้วอาการช้าหายไป จึงชี้ได้ชัดว่าคอขวดหลักอยู่ที่ procedure นี้จริง ไม่ใช่แค่เพราะ Budget ถูกเรียกตอน login แต่เป็นเพราะสิ่งที่ procedure นี้ทำหลังถูกเรียกต่างหาก อย่างไรก็ตาม ประเด็นความเร็วที่ดีขึ้นนี้เป็นคนละเรื่องกับประเด็นข้อมูล budget ฝั่ง project ที่ยังเสี่ยงเพี้ยนจาก query key ที่ผิด
