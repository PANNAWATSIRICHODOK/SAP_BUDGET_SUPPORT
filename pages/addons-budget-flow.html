Overview

ระบบนี้ควบคุมงบของ Draft, PR, PO, A/P Invoice, A/P Credit Memo และ JE

ระบบนี้แบ่งเป็น 2 ด่าน คือ ด่านตรวจข้อมูลต้นทาง ที่เช็กว่าข้อมูลพร้อมหรือยัง และ ด่านคุมงบขั้นสุดท้าย ที่เริ่มจองงบ ตัดงบ และอัปเดตงบคงเหลือจริงก่อนบันทึก

6 เอกสารใน main flow
2 control layers
2 budget ledgers

ถ้าไม่ผ่าน

เอกสารถูก block ได้ทั้งตอนด่านตรวจข้อมูลต้นทาง และตอนด่านคุมงบขั้นสุดท้าย

ถ้าผ่าน

ระบบจะเขียน movement ลง ledger และ recalculation งบคงเหลือให้เอกสารนั้นทันที

สถาปัตยกรรม 2 ชั้น

ชั้นที่ 1: ด่านตรวจข้อมูลต้นทาง

เช็กว่าข้อมูลพร้อมหรือยัง เช่น ฝ่าย ปีงบ และมี budget setup รองรับหรือไม่

ชั้นที่ 2: ด่านคุมงบขั้นสุดท้าย

เริ่มจองงบ ตัดงบ กลับรายการ และอัปเดตงบคงเหลือใน ledger จริง

ผลลัพธ์

เอกสารจะบันทึกได้ก็ต่อเมื่อผ่านทั้งการตรวจข้อมูลต้นทางและการคุมงบขั้นสุดท้าย

ระบบจะเช็ก 3 เรื่องหลัก

ฝ่าย / เจ้าของงบ

เอกสารต้องชี้ให้ชัดว่ารายการนี้เป็นความรับผิดชอบของหน่วยงานไหน

ปีงบ

เอกสารต้องอยู่ในปีงบที่กำหนด ไม่ใช่ข้ามหรือหลุดจากกรอบที่ต้องคุม

มีงบและมียอดคงเหลือ

ชั้นแรกเช็กว่ามีงบรองรับ ส่วนชั้นสุดท้ายจะไปแตะยอดคงเหลือจริงของงบ

เอกสารที่ถูกคุมในภาพรวม
Draft PR PO AP Invoice AP Credit Memo JE

ในด่านคุมงบขั้นสุดท้าย ยังมี logic เพิ่มสำหรับ GRPO และ Goods Return แม้ไม่ได้อยู่ใน main flow presentation นี้

เอกสารแต่ละตัวไปทำอะไรกับงบหลังบ้าน

ส่วนนี้อธิบายเฉพาะเอกสารที่เริ่มขยับงบจริงในด่านคุมงบขั้นสุดท้าย ส่วน Draft และ PR ยังอยู่ฝั่งตรวจข้อมูลต้นทางเป็นหลัก

PO

หน้าที่หลัก: จองงบไว้ก่อนซื้อจริง

เมื่อบันทึกเอกสาร ระบบจะกันวงเงินไว้ให้หน่วยงานหรือโครงการนั้นทันที เพื่อไม่ให้ยอดถูกใช้ซ้ำ

ถ้าเป็นรายการของโครงการและวงเงินไม่พอ เอกสารจะยังไม่ผ่านจนกว่าจะมีงบเพียงพอ

GRPO

หน้าที่หลัก: เปลี่ยนงบที่กันไว้ให้กลายเป็นการใช้จริง

เมื่อรับของจริง ระบบจะย้ายยอดจากส่วนที่กันไว้ ไปเป็นยอดที่ถือว่าเริ่มใช้จริงตามสินค้าที่รับ

จุดนี้ทำให้งบย้ายจาก “กันไว้” ไปเป็น “ใช้จริง”

A/P Invoice

หน้าที่หลัก: รับรู้ค่าใช้จ่ายจริง

ถ้าอ้างจาก PO หรือ GRPO ระบบจะเปลี่ยนยอดที่กันไว้ให้กลายเป็นค่าใช้จ่ายจริงตามยอดเจ้าหนี้ที่บันทึก

ถ้าเอกสารถูกยกเลิก ระบบจะคืนผลกระทบทางงบและคำนวณยอดคงเหลือใหม่ให้อัตโนมัติ

Goods Return

หน้าที่หลัก: ปรับยอดใช้จริงเมื่อมีการคืนของ

เมื่อมีการคืนของ ระบบจะปรับยอดใช้จริงให้สอดคล้องกับปริมาณและมูลค่าที่คืนกลับ

จุดนี้ช่วยให้ยอดใช้งบสะท้อนสถานะของการรับคืนสินค้า

A/P Credit Memo

หน้าที่หลัก: คืนงบจากรายการที่เคยรับรู้ไปแล้ว

ถ้ามีเอกสารอ้างอิง ระบบจะย้อนผลจากรายการเดิม แต่ถ้าไม่มี ระบบจะคืนผลกระทบของงบจากรายการนั้นโดยตรง

จุดนี้ทำหน้าที่คืนงบหรือหักกลับจากรายการที่รับรู้ไปแล้ว

JE

หน้าที่หลัก: ตัดงบตรงจากบัญชี

JE ไม่ได้เริ่มจากการกันวงเงินก่อน แต่จะกระทบงบจริงทันทีหลังผ่านการตรวจข้อมูลต้นทาง

เหมาะกับรายการที่เข้าทางบัญชีโดยตรงและต้องสะท้อนผลในงบเลย

ต้องรีบแก้

6 จุดนี้มีความเสี่ยงทำให้ flow งบผิด, หลุดด่านคุมงบขั้นสุดท้าย, หรือคุม over budget ไม่ตรงที่ตั้งใจ

Urgent NDBS_BUDGET_CONTROL.sql:23-26

JE route ใน wrapper เข้าไม่ถึงตามลำดับเงื่อนไขปัจจุบัน

ปัญหา: มี elseif ( :object_type = '19') ซ้ำสองครั้งทั้ง APCN และ JE

ผลกระทบ: JE branch มีโอกาสไม่ถูก route เข้า NDBS_BUDGET_JE ตามที่ตั้งใจ

ควรแก้: ยืนยัน object type ของ JE แล้วแก้ branch หลังให้เป็นค่าที่ถูกต้อง โดยคาดว่าควรเป็น '30'

High NDBS_BUDGET_CONTROL.sql:31-33

Wrapper ยังเรียก full refresh ทุกครั้งที่ route สำเร็จ

ปัญหา: ตอนจบยังมี call NDBS_UpdateAllBudgetAmount ทุกครั้งที่ error = 0

ผลกระทบ: แม้ helper ใหม่จะเร็วขึ้น แต่ flow ยังผูกอยู่กับ global refresh แทน targeted refresh

ควรแก้: ลด scope ของ recalc หรือเปลี่ยนให้เรียกเฉพาะกรณีที่จำเป็นจริง

High NDBS_BUDGET_AP.sql:604

AP บาง path join budget mapping ผิด key

ปัญหา: ใช้ T5."U_FormatCode" = T4."AcctCode" ทั้งที่จุดใกล้เคียงใช้ FormatCode

ผลกระทบ: budget group อาจถูก map ผิดใน path ที่อ้าง PO เดิม

ควรแก้: ยืนยัน key กลางของ mapping table แล้วใช้ pattern เดียวกันทุก branch

High NDBS_BUDGET_APCN.sql:126-146

APCN cancel path ใช้เงื่อนไขที่จริงเสมอ

ปัญหา: มี IF BCount >= 0 อยู่ 2 จุด

ผลกระทบ: block หลังยกเลิกเอกสารทำงานต่อแม้ไม่มี row ให้ใช้อ้างอิงจริง

ควรแก้: เปลี่ยนเป็น > 0 และ retest cancel ทั้งฝั่ง department และ project

High NDBS_BUDGETNOTIFICATION.sql:385, 505, 821, 941

Notification layer ใช้ account mapping ไม่สม่ำเสมอ

ปัญหา: บาง branch join ด้วย AcctCode แต่บาง branch เปลี่ยนไปใช้ FormatCode

ผลกระทบ: document คนละ path อาจไปเจอ budget group คนละตัว ทั้งที่ business intent เดียวกัน

ควรแก้: เลือก mapping rule กลางเพียงแบบเดียว แล้วใช้ซ้ำทั้งไฟล์

Medium NDBS_BUDGETNOTIFICATION.sql:222, 244, 652, 674

Notification layer ยังมี budget engine ของตัวเอง

ปัญหา: ไฟล์นี้ยัง update @NDBS_BGC_BDPL และ @NDBS_BGC_BPJL โดยตรงหลายจุด

ผลกระทบ: เสี่ยง drift จาก procedure หลัก เพราะมีสองแหล่งคำนวณยอดงบพร้อมกัน

ควรแก้: รวม logic ไว้ที่ helper กลาง หรือเลิกใช้ notification engine ที่ซ้ำซ้อน

เลื่อนลงเพื่อดูว่าแต่ละเอกสารถูกคุมอย่างไรตั้งแต่ด่านต้นทางไปจนถึงด่านคุมงบขั้นสุดท้าย

Workflow

Flow ที่เห็นทั้งความเชื่อมโยงและเหตุผลที่ถูก block

คลิกแต่ละเอกสารเพื่อดูว่าเชื่อมกับขั้นก่อนหน้าและขั้นถัดไปอย่างไร เอกสารถูกตรวจอะไรบ้างก่อนบันทึก และเมื่อผ่านแล้วงบจะเริ่มถูกขยับจริงอย่างไร
