Overview

ระบบ Budget นี้ทำงานเป็น 5 ส่วนที่ต่อกัน

ถ้ามองแบบง่ายที่สุด ระบบจะเริ่มจากการรับ event จาก SAP แล้วตรวจข้อมูลต้นทางก่อน จากนั้นจึงส่งเข้าเครื่องยนต์คุมงบเพื่อจองงบ ใช้งบ คืนงบ และคำนวณยอดคงเหลือกลับเข้า SAP Business One

แต่ละส่วนใช้ทำอะไร
ด่านรับงาน

SBO_SP_TransactionNotification

รับ event จาก SAP แล้วเช็กข้อมูลต้นทางว่าเอกสารพร้อมเข้าสู่การคุมงบหรือยัง

เครื่องยนต์หลัก

NDBS_BUDGET_CONTROL

เลือกเส้นทางตามชนิดเอกสาร แล้วเริ่มจองงบ ใช้งบ คืนงบ หรือย้อนผลกระทบของงบ

เส้นทาง PR

NDBS_BUDGET_PR

เป็นทางแยกเฉพาะของ PR ใช้จัดการรายการต้นทางของคำขอใช้งบ

ตัวคำนวณยอด

NDBS_UpdateBudgetAmount

รวบรวมยอดที่กันไว้ ยอดที่ใช้จริง และอัปเดตยอดคงเหลือกลับเข้า master ของงบ

เส้นสร้างย้อนหลัง

NDBS_BUDGET_BEGINING

ใช้สร้าง movement ย้อนหลังจากเอกสารเดิม เมื่อต้อง rebuild ระบบงบใหม่ทั้งชุด

ระบบนี้ทำงานอยู่ 2 แบบ

Runtime Flow

ใช้ตอนผู้ใช้เพิ่ม แก้ไข ยกเลิก หรือปิดเอกสารจริงใน SAP แล้วระบบคุมงบทันทีใน transaction นั้น

Rebuild Flow

ใช้ตอนต้องสร้าง movement งบย้อนหลังจากเอกสารเดิม เพื่อให้ ledger กับยอดคงเหลือกลับมาตรงกันใหม่

เอกสารที่อยู่ในวงจร

Draft, PR, PO, GRPO, A/P Invoice, A/P Credit Memo และ JE แต่ละตัวมีผลต่องบไม่เหมือนกัน

สรุปการทำงานแบบสั้นที่สุด

1. ตรวจข้อมูลก่อน

ระบบเช็กฝ่าย ปีงบ และ budget setup ก่อน ถ้าไม่พร้อมจะ block ตั้งแต่ต้นทาง

2. ขยับงบตามเอกสาร

เมื่อผ่านแล้ว engine จะเลือกว่าควรกันงบ ใช้งบ คืนงบ หรือย้อนรายการเดิมตามชนิดเอกสาร

3. อัปเดตยอดคงเหลือ

ทุกครั้งที่ movement เปลี่ยน ระบบจะคำนวณยอดคงเหลือใหม่ก่อนปล่อยให้ transaction จบ

เอกสารแต่ละตัวไปทำอะไรกับงบหลังบ้าน

ส่วนนี้เป็นเวอร์ชันผสมระหว่างคำอธิบายแบบสั้นที่เคยใช้ตอนเพิ่ม NDBS_BUDGET_CONTROL กับรายละเอียดจาก SQL ปัจจุบัน เพื่อให้ผู้ใช้งาน SAP B1 เข้าใจง่าย แต่ยังไม่เสียภาพจริงของระบบ

Draft

หน้าที่หลัก: เช็กความพร้อมก่อนออกเอกสารจริง

ระบบเช็กก่อนผ่าน: ตรวจว่าบรรทัดที่เกี่ยวกับงบใส่ฝ่ายหรือยัง, ใส่ปีงบหรือยัง, และสำหรับ draft ของ PR หรือ PO จะเช็กต่อว่ามีงบรองรับทั้งกรณีตามฝ่ายและตามโครงการหรือไม่

ผ่านแล้วระบบทำอะไร: อนุญาตให้ draft ถูกนำไปเปิดต่อเป็นเอกสารจริง เช่น PR, PO, A/P Invoice หรือ A/P Credit Memo ตาม flow งาน

ผู้ใช้ควรเข้าใจว่า: Draft ยังไม่กระทบงบเลย ยังไม่จองงบ ยังไม่ใช้จริง และยังไม่คืนงบ เป็นเพียงด่านตรวจข้อมูลเท่านั้น

SQL ที่เกี่ยวข้อง: กดดูโค้ดจริงของ draft

PR

หน้าที่หลัก: เป็นคำขอใช้งบก่อนออก PO

ระบบเช็กก่อนผ่าน: ตรวจว่าบรรทัด PR ใส่ฝ่ายหรือยัง, ใส่ปีงบหรือยัง, ถ้าไม่มีโครงการจะต้องมีงบของฝ่ายรองรับ และถ้ามีโครงการจะต้องมีงบของโครงการรองรับด้วย

ผ่านแล้วระบบทำอะไร: ในมุมผู้ใช้งาน PR จะพร้อมเข้าสู่ approval และถูกใช้เป็นฐานของ PO ต่อไป

หมายเหตุสำคัญ: ใน SQL ปัจจุบันมี procedure แยกสำหรับให้ PR ไปลงผลกับงบต่อ แต่ route runtime ยังมีจุดที่ต้องแก้ ดังนั้นตอนนี้ควรมองว่า PR อยู่ฝั่งเช็กข้อมูลต้นทางเป็นหลัก

ผู้ใช้ควรเข้าใจว่า: PR คือคำขอใช้เงินต้นทาง ยังไม่ใช่การใช้เงินจริงแบบ PO แต่เป็นจุดที่ระบบบังคับให้ข้อมูล budget ต้องครบก่อนเริ่มจัดซื้อจริง

SQL ที่เกี่ยวข้อง: กดดูโค้ดจริงของ PR

PO

หน้าที่หลัก: จองวงเงินไว้ก่อนซื้อจริง

ระบบเช็กก่อนผ่าน: ตรวจว่าบรรทัด PO ใส่ฝ่ายหรือยัง, ใส่ปีงบหรือยัง, มีงบรองรับหรือไม่ทั้งกรณีตามฝ่ายและตามโครงการ และยังเช็กรูปแบบของปีงบอีกชั้นหนึ่ง

ผ่านแล้วระบบทำอะไร: ระบบจะจองวงเงินของฝ่ายหรือโครงการที่ระบุไว้ทันที เพื่อไม่ให้วงเงินก้อนเดียวกันถูกใช้ซ้ำโดยเอกสารอื่น

ถ้างบไม่พอหรือมีการเปลี่ยนแปลง: ถ้าเป็นรายการของโครงการและวงเงินไม่พอ เอกสารอาจไม่ผ่านในด่านคุมงบขั้นสุดท้าย และถ้าแก้ไข ยกเลิก หรือปิด PO ระบบจะย้อนรายการเดิมและคืนวงเงินส่วนที่ยังไม่ใช้กลับมาให้อัตโนมัติ

ผู้ใช้ควรเข้าใจว่า: PO ยังไม่ใช่ค่าใช้จ่ายจริง แต่เป็นสถานะ “กันไว้รอซื้อ” เพื่อจองงบก่อนถึงขั้นรับของหรือบันทึกเจ้าหนี้

SQL ที่เกี่ยวข้อง: กดดูโค้ดจริงของ PO

GRPO

หน้าที่หลัก: เปลี่ยนจากจองงบเป็นเริ่มใช้จริง

ระบบใช้ข้อมูลอะไรเป็นฐาน: GRPO จะอิงข้อมูลจากเอกสารต้นทางเดิม โดยเฉพาะ PO ที่เคยจองงบไว้ แล้วอ่านข้อมูลฝ่ายหรือโครงการจากรายการที่รับของนั้น

ผ่านแล้วระบบทำอะไร: เมื่อรับของจาก PO ระบบจะย้ายยอดจากส่วนที่จองไว้ ไปเป็นยอดที่ถือว่าเริ่มใช้จริงตามมูลค่าของที่รับเข้า

ถ้ามีการยกเลิก: ระบบจะย้อนผลของการรับของกลับ และคำนวณยอดคงเหลือใหม่ให้สอดคล้องกับสถานะล่าสุด

ผู้ใช้ควรเข้าใจว่า: GRPO คือจุดที่งบขยับจาก “จองไว้” ไปเป็น “ใช้จริง” เพราะมีการรับของเข้ามาแล้ว

SQL ที่เกี่ยวข้อง: กดดูโค้ดจริงของ GRPO

A/P Invoice

หน้าที่หลัก: ยืนยันค่าใช้จ่ายจริงทางเจ้าหนี้

ระบบเช็กก่อนผ่าน: ตรวจว่ามีปีงบหรือยัง และปีงบอยู่ในรูปแบบที่ระบบยอมรับหรือไม่ ก่อนจะให้เอกสารเข้าไปกระทบงบ

ผ่านแล้วระบบทำอะไร: เมื่อบันทึก A/P Invoice ระบบจะถือว่ารายการนั้นเป็นค่าใช้จ่ายจริงแล้ว ไม่ว่าจะอ้างจาก PO, GRPO หรือบันทึกตรงตามเงื่อนไขที่โค้ดรองรับ

ถ้าอ้างอิงเอกสารเดิมหรือมีการยกเลิก: ถ้าอ้างจาก PO หรือ GRPO ระบบจะเปลี่ยนยอดที่จองไว้หรือยอดที่เริ่มใช้จริง ให้กลายเป็นค่าใช้จ่ายจริงตามยอดเจ้าหนี้ที่บันทึก และถ้ายกเลิกเอกสาร ระบบจะคืนผลกระทบของงบและคำนวณยอดใหม่ให้อัตโนมัติ

ผู้ใช้ควรเข้าใจว่า: A/P Invoice คือจุดที่ค่าใช้จ่ายถูกยืนยันชัดเจนในงบและผูกกับภาระเจ้าหนี้จริง

SQL ที่เกี่ยวข้อง: กดดูโค้ดจริงของ A/P Invoice

Goods Return

หน้าที่หลัก: ลดการใช้จริงเมื่อมีการคืนของ

ระบบใช้ข้อมูลอะไรเป็นฐาน: Goods Return จะอิงรายการของที่เคยรับเข้าหรือเคยใช้งบไปแล้ว เพื่อรู้ว่าต้องลดผลกระทบของงบจากรายการใด

ผ่านแล้วระบบทำอะไร: เมื่อมีการคืนของ ระบบจะปรับยอดใช้จริงให้ลดลงตามปริมาณและมูลค่าที่คืนกลับ เพื่อไม่ให้ใช้งบค้างเกินจากของที่ถือไว้จริง

ผู้ใช้ควรเข้าใจว่า: Goods Return ช่วยดึงยอดใช้งบกลับลงบางส่วน ทำให้งบสะท้อนสถานะของการรับคืนสินค้าได้ถูกต้องขึ้น

SQL ที่เกี่ยวข้อง: กดดูโค้ดจริงของ Goods Return

A/P Credit Memo

หน้าที่หลัก: คืนค่าใช้จ่ายหรือคืนงบ

ระบบเช็กก่อนผ่าน: ตรวจว่ามีปีงบหรือยัง ก่อนให้เอกสารเข้าไปคืนผลกระทบของงบ

ผ่านแล้วระบบทำอะไร: เมื่อมี A/P Credit Memo ระบบจะมองว่าเราได้รับการลดหนี้หรือย้อนค่าใช้จ่ายบางส่วนกลับมา จึงต้องคืนผลกระทบของงบที่เคยรับรู้ไว้

ถ้ามีหรือไม่มีเอกสารอ้างอิง: ถ้ามีเอกสารอ้างอิง ระบบจะย้อนผลจากรายการเดิมก่อน แต่ถ้าไม่มี ระบบจะคืนผลกระทบของงบจากเอกสารนั้นโดยตรง โดยในฝั่งโครงการยังมีเงื่อนไขใน SQL ที่ควรแก้ให้ชัดเจนอีกจุดหนึ่ง

ผู้ใช้ควรเข้าใจว่า: A/P Credit Memo คือเอกสารที่ดึงค่าใช้จ่ายกลับ และทำให้งบถูกคืนตามรายการที่เคยรับรู้ไปแล้ว

SQL ที่เกี่ยวข้อง: กดดูโค้ดจริงของ A/P Credit Memo

JE

หน้าที่หลัก: กระทบงบตรงจากรายการบัญชี

ระบบเช็กก่อนผ่าน: ตรวจว่ามีปีงบหรือยัง และปีงบอยู่ในรูปแบบที่ระบบยอมรับหรือไม่ แม้จะไม่ได้เช็ก budget setup แบบ PR หรือ PO

ผ่านแล้วระบบทำอะไร: JE ไม่ผ่านขั้นจองวงเงินแบบ PO แต่ถ้าผ่านการตรวจข้อมูลแล้ว ระบบจะถือว่ารายการนั้นกระทบงบจริงทันที

ถ้ามีการแก้ไข: ระบบจะอัปเดตผลกระทบของงบตามข้อมูลล่าสุดของ JE เพื่อให้ยอดคงเหลือไม่ค้างตามรายการเดิม

ผู้ใช้ควรเข้าใจว่า: เหมาะกับรายการที่เข้าทางบัญชีโดยตรงและต้องสะท้อนผลในงบทันที โดยไม่ต้องรอผ่านเอกสารจัดซื้อก่อน

SQL ที่เกี่ยวข้อง: กดดูโค้ดจริงของ JE

Rebuild ย้อนหลัง

หน้าที่หลัก: สร้างผลกระทบงบย้อนหลังจากเอกสารเดิม

ระบบใช้ข้อมูลอะไรเป็นฐาน: อ่านเอกสารเก่าที่มีอยู่ในระบบ เช่น PO, GRPO, A/P Invoice, Return, A/P Credit Memo และ JE เพื่อสร้างประวัติงบย้อนหลังขึ้นมาใหม่

เมื่อรันแล้วระบบทำอะไร: จะไล่เอกสารทีละประเภทแล้วสร้างผลกระทบงบย้อนหลัง พร้อมคำนวณยอดใหม่ทั้งชุดให้ข้อมูลปัจจุบันกลับมาตรงกับเอกสารที่เคยเกิดไปแล้ว

ข้อควรระวัง: เส้นนี้ไม่ใช่งานประจำวันที่ผู้ใช้ SAP B1 ทำทุกวัน แต่เป็นเครื่องมือซ่อมระบบงบ และตอนนี้ PR ในเส้นนี้ยังไม่ได้เปิดใช้งานจริง

SQL ที่เกี่ยวข้อง: กดดูโค้ดจริงของ rebuild
